# Provenance Guard

A backend any creative-sharing platform can plug in to classify whether submitted
text was written by a human or generated by AI — then surface an honest,
plain-language transparency label, accept creator appeals, and keep a structured
audit trail. Built to be **conservative**: on a writing platform, falsely
branding a human's work as AI is the worst outcome, and the design reflects that.

> Full design rationale lives in [planning.md](planning.md) (written before any code).

## Stack

| Component | Tool |
|-----------|------|
| API | Flask |
| Signal 1 (semantic) | Groq `llama-3.3-70b-versatile` |
| Signal 2 (structural) | Stylometric heuristics (pure Python) |
| Rate limiting | Flask-Limiter |
| Storage / audit log | SQLite |

## Run it

```bash
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env               # add your GROQ_API_KEY
PORT=5001 python -m src.app        # port 5001 avoids macOS AirPlay on 5000
```

Then open **http://localhost:5001/** for the web UI: paste text → Analyze →
color-coded transparency label (red AI / green human / amber uncertain) with a
confidence bar and per-signal breakdown, an appeal box, and a live audit-log
table. The UI is a single vanilla-JS page ([static/index.html](static/index.html))
served same-origin by Flask — no build step, no framework.

Offline self-check (no API key needed — runs the stylometry-only path):

```bash
python -m tests.test_pipeline
```

## Architecture overview — the path a submission takes

`POST /submit` validates `{text, creator_id}`, then fans the text out to **two
independent signals**. **Signal 1 (Groq LLM)** judges meaning/voice; **Signal 2
(stylometry)** measures surface statistics. Each returns `P(AI) ∈ [0,1]`. The
**scorer** blends them — `p_ai = 0.65·llm + 0.35·stylo` — and applies
**asymmetric thresholds** to pick an attribution (`likely_ai` / `uncertain` /
`likely_human`) plus a `confidence`. The **label builder** turns that into
reader-facing text. The result is written to the `submissions` table and an
entry is appended to `audit_log`, and the JSON response goes back to the client.
An **appeal** (`POST /appeal`) flips the submission to `under_review` and appends
an appeal event carrying the original decision + the creator's reasoning. See the
diagram in [planning.md](planning.md#architecture).

## Detection signals — what each measures, why, and what it misses

**Signal 1 — Groq LLM (semantic).** Asks the model for the probability the text
is AI-generated, judging holistic coherence, originality of phrasing, and "voice."
*Why:* AI text is often fluent-but-generic in ways a strong model recognizes
holistically — something no surface metric captures. *Blind spots:* can be wrong
on very short, non-English, or jargon-heavy text; costs a network call; can be
gamed by paraphrasing. Degrades gracefully — if `GROQ_API_KEY` is missing or the
call fails, the system falls back to stylometry alone.

**Signal 2 — Stylometry (structural).** Pure-Python, three interpretable metrics:
- **Burstiness** (coefficient of variation of sentence length) — human prose
  mixes long and short sentences; AI prose is uniform. *Primary, weight 0.55.*
- **Type-token ratio** (vocabulary diversity) — mid-range is AI-typical; very
  low or very high skews human. Scored with a gaussian centered at 0.5. *0.25.*
- **Expressive punctuation density** (`! ? ... --`) — sparse/even → AI. *0.20.*
- **Casual-marker discount** (`i`, `ok`, `honestly`, `lol`…) pulls toward human,
  protecting informal writers.

**Signal 3 — Input provenance (behavioral, stretch/ensemble, optional).** Reads
*how* the text entered the box, not its content: keystrokes, backspaces, paste
events, and elapsed time, captured in the browser. *Typed live with corrections*
→ strong human provenance; *pasted as a block* → **weak** provenance. Crucially,
**paste ≠ AI** — people paste their own drafts — so a paste only nudges suspicion
up; the score is clamped to [0.10, 0.65] and capped at 0.25 weight, never able to
force an AI verdict alone. API/curl callers omit it and it drops out cleanly.
*Blind spots:* only available in the browser UI; a determined faker could retype
AI text (which is exactly why it's weighted low and paired with content signals).

*Why pair them:* each reads a different axis — meaning, form, origin-of-entry —
genuinely independent, so text that fools one often won't fool the others.
*Stylometry blind spots:* short text has no sentence variance (looks AI); formal
human writing is uniform (looks AI); it measures form, never truth.

## Confidence scoring — how it's combined, validated, and what it means

**Ensemble weighting:** base weights `llm 0.50 · stylo 0.25 · behavior 0.25`.
Any unavailable signal is dropped and the rest **renormalized** — so with no
behavioral data (API/curl) it reduces to `0.667 · llm + 0.333 · stylo`, and
`p_ai = Σ(weightᵢ·scoreᵢ)/Σweightᵢ`. The LLM carries the most weight because it
reads meaning; behavioral provenance is capped low because paste ≠ AI.
Attribution uses **asymmetric thresholds** so we don't accuse humans on weak
evidence:

| `p_ai` | Attribution |
|--------|-------------|
| ≥ 0.70 | `likely_ai` |
| 0.40 – 0.70 | `uncertain` |
| ≤ 0.40 | `likely_human` |

`confidence = max(p_ai, 1 − p_ai)` ∈ [0.5, 1.0] — what the label renders, so a
`p_ai` of 0.51 and 0.95 produce **visibly different labels**, not a binary flip.

**How I validated it's meaningful:** `tests/test_pipeline.py` asserts that
clearly-AI text scores a higher `p_ai` than clearly-human text by a margin
(> 0.15) and that confidence is always well-formed. I also ran four deliberately
chosen inputs (clear-AI, clear-human, two borderline) and checked the scores
matched intuition; where they didn't, I printed each signal separately to find
which one was off (see AI usage below).

**Real example submissions (both signals live), spanning all three bands:**

| Input | LLM | stylo | `p_ai` | confidence | attribution |
|-------|-----|-------|--------|-----------|-------------|
| "In conclusion, it is important to recognize that effective time management… Ultimately, these strategies contribute to a balanced and successful lifestyle." | 0.80 | 0.55 | **0.713** | **0.713** | `likely_ai` |
| "Artificial intelligence represents a transformative paradigm shift… stakeholders must collaborate to ensure responsible deployment." | 0.80 | 0.44 | **0.673** | **0.673** | `uncertain` |
| "ok so i finally tried that new ramen place downtown and honestly? underwhelming… probably wont go back unless someone drags me there" | 0.20 | 0.00 | **0.130** | **0.870** | `likely_human` |

The middle row is the multi-signal design working as intended: the LLM reads the
formal essay as AI (0.80) while stylometry, seeing uniform-but-not-extreme prose,
stays moderate (0.44). They disagree, so the system reports **uncertain** rather
than a false-confident verdict — exactly the honesty the project asks for.

## Transparency label — three variants (exact text)

`{pct}` = `round(confidence × 100)`, injected at render time.

| Variant | Exact label text |
|---------|------------------|
| **High-confidence AI** | 🤖 Likely AI-generated — Our analysis suggests this content was probably created with the help of AI ({pct}% confidence). This is an automated estimate, not a certainty — AI detection is imperfect. If you wrote this yourself, you can appeal. |
| **High-confidence human** | ✍️ Likely human-written — Our analysis suggests this content was probably written by a person ({pct}% confidence). This is an automated estimate and is not a guarantee. |
| **Uncertain** | ❓ Uncertain origin — We couldn't confidently determine whether this content was written by a person or generated by AI (only {pct}% certainty either way). We're showing this honestly rather than guessing. Treat the origin as unknown. |

Design intent: plain language, no jargon, every variant states it's an *estimate*,
and the AI variant always points to the appeal path — the false-positive backstop.

## Appeals workflow

`POST /appeal` with `{content_id, creator_reasoning}`:
1. Looks up the submission (404 if unknown `content_id`).
2. Sets its status to **`under_review`**.
3. Appends an `appeal` audit event carrying **both** the original decision (so a
   reviewer sees what was contested) and the creator's reasoning.
4. Returns confirmation. No automated re-classification — a human owns the call.

```json
{
  "content_id": "4299bfe4-...",
  "status": "under_review",
  "message": "Appeal received. A human reviewer will re-examine this classification.",
  "original_attribution": "likely_human",
  "original_confidence": 1.0
}
```

## Rate limiting

Applied to `POST /submit`: **`10 per minute; 100 per day`** per IP
(`/appeal` is capped at `20 per hour`).

**Reasoning:** a real creator submits their own work occasionally — a handful of
pieces in a sitting, not dozens per second. 10/min comfortably covers genuine
bursts (revising and re-checking a draft) while stopping a script from flooding
the classifier (each `/submit` triggers a paid LLM call, so the cap is also cost
control). 100/day bounds sustained abuse from a single source without throttling
any plausible human. These are per-IP via `get_remote_address`; a production
deployment would also key on authenticated account.

**Evidence** — 12 rapid requests, first 8 succeed (2 earlier submits already
counted toward the 10/min window), rest are rejected:

```

```

## Stretch implemented — Ensemble detection (3 signals)

A third signal, **input provenance**, is layered on top of the two content
signals and combined with the documented renormalized weighting above. The
browser records keystrokes, backspaces, paste events, and elapsed time; the
backend turns that into `P(AI)` and a verdict (`typed_live` / `mixed` /
`pasted_block`), shown in the UI as a badge (⌨️ Typed live / 📋 Pasted block).

It is **weak by design** — paste is not proof of AI — so it nudges, never
dominates (clamped to [0.10, 0.65], weight ≤ 0.25). Demonstration with the
*same* formal-essay text (LLM reads it as AI, stylo moderate):

| How it was entered | behavior score | `p_ai` | attribution |
|--------------------|---------------|--------|-------------|
| **Typed live** (230 keystrokes, 14 corrections, 48 s) | 0.10 | **0.588** | `uncertain` |
| **Pasted block** (one 210-char paste, 0 keystrokes) | 0.63 | **0.720** | `likely_ai` |
| **API / curl** (no behavioral data) | n/a | 0.750 | `likely_ai` |

This is also the fix for the most common false positive: clean, carefully *typed*
human prose used to read as AI on content alone; the typing evidence now pulls it
back toward human. It still won't claim "human" outright when the content itself
reads as AI — it correctly lands on `uncertain`, which is the honest answer.

## Known limitations

- **Very short text is mislabeled as AI.** A single sentence has no
  sentence-length variance (CV = 0), so the burstiness metric — stylometry's
  primary input — reads "uniform" and pushes `p_ai` up. The 8-word rate-limit
  test strings landed at `likely_ai` (0.751) for exactly this reason. This is a
  property of the stylometric signal, not a tuning issue; the LLM weight and the
  appeal path are the mitigations, and very short submissions are flagged in
  planning.md as out-of-comfort-zone.
- **Formal human writing** (academic/legal/financial) is naturally uniform and
  can lean AI — which is precisely why the AI threshold is a high 0.70.

## Spec reflection

- **Helped:** writing the three label variants *as exact strings* in planning.md
  before coding meant `build_label` was a near-mechanical translation, and the
  thresholds (0.70 / 0.40) were decided in the spec — so scoring had a concrete
  target instead of being reverse-engineered from code.
- **Diverged:** the spec implied port 5000; macOS reserves it for AirPlay, so the
  app reads a `PORT` env var (default 5001). I also added a `/health` endpoint
  and graceful LLM degradation (stylometry-only when no key) that the spec didn't
  call for, because they made the system testable offline and in CI.

## AI usage

1. **Flask skeleton + first signal (M3).** I gave the AI the detection-signals
   section and the diagram and asked for the `/submit` route plus `llm_signal`.
   It returned a workable skeleton, but I overrode the prompt to force a strict
   JSON response (`response_format=json_object`, `temperature=0`) and added the
   no-key fallback so the signal degrades instead of crashing.
2. **Stylometric scoring (M4).** I asked for the three-metric `stylo_signal`. The
   first version weighted TTR linearly (higher TTR = more AI), which mislabeled
   casual high-vocabulary human text. I revised it to a gaussian-centered TTR and
   added the casual-marker discount after printing each signal separately on the
   four test inputs and seeing the human-ramen sample score too high.
